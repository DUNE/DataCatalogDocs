Notes on the SAM data catalog system

Â 

H. Schellman 1-28-2020 - updated Dec 2021

Â 

SAM is a data catalog originally designed for the D0 and CDF high energy physics experiments at Fermilab.Â  It is now used by most of the Intensity Frontier experiments at Fermilab.

Â 

The most important objects cataloged in SAM are individual files and collections of files called datasets.

Â 

Data files themselves are not stored in SAM, their metadata is, and that metadata allows you to search for and find the actual physical files.Â  Sam also provides mechanisms for initiating and tracking file delivery through projects.

Â 

General considerations

Â 

SAM was designed to ensure that large scale data-processing was done completely and accurately which leads to some features not always present in a generic catalog but very desirable if one wishes high standards of reproducibility and documentation in data analysis.

Â 

For example, at the time of the original design, the main storage medium was 8mm tapes using consumer-grade drives.Â  Drive and tape failure rates were > 1%.Â  Several SAM design concepts, notably luminosity blocks and parentage tracking, were introduced to allow accurate tracking of files and their associated normalization in a high error-rate environment.

Â 

Â·Â Â Â Â Â  Description of the contents of data collections to allow later retrieval

Â 

Â·Â Â Â Â Â  Tracking of object and collection parentage and description of processing transformations to document the full provenance of any data object and ensure accurate normalization

Â 

Â·Â Â Â Â Â  Grouping of objects and collection into larger âdatasetsâ based on their characteristics

Â 

Â·Â Â Â Â Â  Storing physical location of objects

Â 

Â·Â Â Â Â Â  Tracking of the processing of collections to allow reprocessing on failure and avoid double processing.

Â 

Â·Â Â Â Â Â  Methods (âprojectsâ) for delivering and tracking collections in multi-processing jobs

Â 

Â·Â Â Â Â Â  Preservation of data about processing/storage for debugging/reporting

Â 

The first 3 goals relate to content and characteristics while the last 3 relate to data storage and processing tools.Â

Specifics

Â 

1.Â Â Â Â  The current SAM implementation uses the file as the basic unit of information.Â  Metadata is associated with the file name.Â  Filenames must be unique in the system. This prevents duplication of data in a sample, as a second copy cannot be cataloged. This makes renaming a file very unwise. A very common practice is to include some of the metadata in the filename, both to make it easier to identify and to ensure uniqueness.Â Â Â 

Â 

2.Â Â  Â Â Metadata for a file can include file locations but does not have to. A file can have no location at all, or many. When you move or remove a file with an associated SAM location, you need to update the location information.Â 

 

3.Â Â Â Â  SAM does not move files..Â  It provides location information for a process to use in streaming or copying a file using its own methods. Temporary locations (such as on a grid node) need not be reported to SAM. Permanently storing or removing files requires both moving/removing the file itself and updating its metadata to reflect that location and is generally left up to special packages such as the Fermilab FTS (File Transfer Service) and SAM projects.

Â 

4.Â Â Â Â  Files which are stored on disk or tape are expected to have appropriate file sizes and checksums. One can have duplicate instances of a file in different locations, but they must all be identical.Â  If one reprocesses an input file, the output may well be subtly different (for example dates stored in the file itself can change the checksum). SAM should force you to choose which version, old or new, is acceptable.Â  It will not let you catalog both with the same filename. As a result, if you get a named file out of SAM, you can be reasonably certain you got the right copy.

Â 

5.Â Â Â Â  files with duplicate content but different names can be problematic. The reprocessed file mentioned in part 4, if renamed, could cause significant problems if it were allowed into the data sample along with the originals as a job processing all files might get both copies. This is one of the major reasons for the checksums and unique filenames. There is a temptation to put, for example, timestamps, in filenames to generate unique names but that removes a protection against duplication.

Â 

6.Â Â Â Â  Files can have parents and children and, effectively, birth certificates that can tell you how they were made.Â  An example would be a set of raw data files RAWnnn processed with code X to produce a single reconstructed file RECO.Â  One can tell SAM that RAWnnn are the parents of RECO processed with version x of code X. If one later finds another RAWnnn file that was missed in processing, SAM can tell you it has not been processed yet with X (i.e., it has no children associated with version x of X) and you can then choose to process that file. This use case often occurs when a production job fails without reporting back or succeeds but the copy back or catalog action fails.

Â 

The D0 experiment required that all official processing going into SAM be done with tagged releases and fixed parameter sets to increase reproducibility and the tags for that information were included in the metadata. Calibration databases were harder to timestamp so some variability was still possible if calibrations were updated.

Â 

7.Â  SAM supports several types of data description fields:

Â 

Values are standard across all implementations like run_type, file_size â¦

Â 

Parameters are defined by the experiment for example MC.Genieversion

Â 

Values are common to almost allÂ  HEP experiments and are optimized for efficient queries.Â  SAM also allows definition of âparametersâ (by administrators) as they are needed.Â  This allows the schema to be modified easily as needs arrive.

Â 

8.Â  Metadata can also contain âspillâ or luminosity block information that allows a file to point to specific data taking periods with smaller granularity than a run or subrun. When files are merged, this spill information is also merged.

Â 

9. SAM currently does not contain a means of easily determining which file a given event is in.Â  If a daq system is writing multiple streams, an event from a given subrun could be in any stream.Â  Adding an event database would be a useful feature.

Â 

All of these features are intended to assure that your data are well described and can be found. As SAM stores full location information, this means any SAM-visible location. In addition, if parentage information is provided, you can determine and reproduce the full provenance of any file.

Datasets and projects

Â 

Datasets

Â 

In addition to the files themselves, SAM allows you to define datasets.

Â 

A SAM dataset is not a fixed list of files but a query against the SAM database. An example would be âdata_tier reconstructed and run_number 2001 and version v10â which would be all files from run 2001 that are reconstructed data produced by version v10. This dataset is dynamic. If one finds a missing file from run 2001 and reconstructs it with v10, the dataset will grow. There are also dataset snapshots that are derived from datasets and capture the exact files in the dataset when the snapshot was made.

Note: most other data catalogs assume a âdatasetâ is a fixed list of files.Â  This is a âsnapshotâ in SAM.

Projects

Â 

Â 

SAM also supports access tracking mechanisms called projects and consumers.Â  These are generally implemented for you by grid processing scripts. Your job is to choose a dataset and then ask the processing system to launch a project for that dataset.

Â 

A project is effectively a processing campaign across a dataset which is owned by the SAM system. At launch a snapshot is generated and then the files in the snapshot are delivered to a set of consumers.Â  The project maintains an internal record of the status of the files and consumers. Each grid process can instantiate a consumer which is attached to the project.Â  Those consumers then request âfilesâ from the project and, when done processing, tell the project of their status.Â 

Â 

The original SAM implementation actually delivered the files to local hard drives.Â  Modern SAM delivers the location information and expects the consumer to find the optimal delivery method. This is a pull model, where the consuming process requests the next file rather than having the file assigned to it.Â  This makes the system more robust on distributed systems.

Checking the database does not require special privileges but storing files and

Suggestions for configuring SAM (for admins)

Â 

First of all, it really is nice to have filenames and dataset names that tell you whatâs in the box, although not required. The D0 and MINERvA conventions have been to use â_â underscores between useful key strings. As a result, D0 and MINERvA tried not to use â_â in metadata entries to allow cleaner parsing. â-â is used if needed in the metadata.

Â 

D0 also appended processing information to filenames as they moved through the system to assure that files run through different sequences had unique identifiers.

Â 

Example: A Monte Carlo simulation file generated with version v3 and then reconstructed with v5 might look like

Â 

SIM_MC_020000_0000_simv3.root would be a parent of RECO_MC_020000_0000_simv3_recov5.root

Data files are all children of the raw data while simulation files sometimes have more

complicated ancestry, with both unique generated events and overlay events from data as parents.

Â 

Setting up SAM metadata (For admins)

Â 

This needs to be done once, and very carefully, early in the experiment. It can grow but thinking hard at the beginning saves a lot of pain later.

Â 

You need to define data_tiers. These represent the different types of data that you produced through your processing chain. Examples would be raw, pedsup, calibrated, reconstructed, thumbnail, mc-generated, mc-geant, mc-overlaid,

Â 

run_type can be used to support multiple DAQ instances.

Â 

data_stream is often used for trigger subsamples that you may wish to split data into (for example pedestal vs data runs).

Â 

Generally, you want to store data from a given data_tier with other data from that tier to facilitate fast sequential access.

Â 

Applications

Â 

It is useful, but not required to also define applications which are triads of âappfamilyâ, âappnameâ and âversionâ. Those are used to figure out what changed X to Y. There are also places to store the machine the application ran on and the start and end time for the job.

 

The query:

Â 

samweb list-files "data_tier raw and not isparentof: (data_tier reconstructed and appname reco and version 7)"

Â 

Should, in principle, list raw data files not yet processed by version 7 of reco to produce files of tier reconstructed. You would use this to find lost files in your reconstruction after a power outage.

Â 

It is good practice to also store the name of the head application configuration file for processing but this does not have a standard âvalueâ

Merging and splitting (for experts)

Â 

Parentage works pretty well if one is merging files but splitting them can become problematic as it makes the parentage structure pretty complex.

Sam will let you merge files with different attributes if you donât check carefully. Generally, it is a good idea not to merge files from different data tiers and certainly not from different data_types. Merging across major processing versions should also be avoided.
